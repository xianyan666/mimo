# review-checklist

## 1. 审查结论

**WARNING**：代码整体可正常运行，核心功能（submit/today/check/review）均已实现且测试通过。但存在以下需要关注的问题：ID 生成存在碰撞风险、缓存机制可能导致数据不一致、部分异常场景未覆盖、代码存在重复。这些问题不影响基本使用，但在高频操作或异常情况下可能导致数据问题，建议尽快修复。

## 2. 风险发现表

| 编号 | 风险类型 | 位置 | 问题描述 | 影响 | 修复方案 | 对应测试 |
|------|----------|------|----------|------|----------|----------|
| R1 | 功能正确性风险 | `models/record.py:61-65` `generate_id()` 函数 | ID 生成使用 `datetime.now().strftime("%Y%m%d%H%M%S")` + `random.randint(100, 999)`，同一秒内提交多条记录时，3 位随机数（100-999，共 900 种可能）存在碰撞概率 | 同一秒内多次提交可能导致 ID 重复，破坏记录唯一性 | 改用 `uuid.uuid4()` 或在时间戳后追加更长的随机数（如 6 位）或使用自增序列 | 编写循环 1000 次调用 `generate_id()` 的测试，断言所有 ID 唯一 |
| R2 | 数据与状态风险 | `storage/memory_store.py:14-18` `_load()` 方法 | `_data` 缓存只在首次加载时读取文件，后续调用直接返回缓存。如果文件被外部修改（如手动编辑、多进程写入），缓存数据不会更新 | 多进程场景下数据不一致；手动修复数据文件后程序仍使用旧缓存 | 方案一：移除缓存，每次 `_load()` 都读取文件；方案二：添加 `_invalidate_cache()` 方法，在 `_save()` 后清除缓存 | 测试：保存记录后，手动修改文件内容，再次读取验证是否获取最新数据 |
| R3 | 异常处理风险 | `core/analyzer.py:105` 和 `core/weakness.py:66` | `key.split(":")` 假设知识点 key 格式为 `subject:chapter`，但如果章节名称中包含冒号（如 "I/O: 输入输出"），会导致解析错误 | 章节名含冒号时程序崩溃或数据解析错误 | 改用更安全的分隔符（如 `|||`），或使用 `key.split(":", 1)` 限制分割次数为 1 | 构造章节名包含冒号的测试数据，验证解析是否正确 |
| R4 | 功能正确性风险 | `commands/submit.py:10-15` `SUBJECTS` 字典 | `SUBJECTS` 字典硬编码了科目映射 `{"1": "数据结构", ...}`，与 `ConfigStore` 中的科目列表独立。如果用户通过 config 修改了科目，submit 命令仍使用硬编码映射 | 科目配置与实际提交的科目不一致，可能导致数据混乱 | 移除硬编码的 `SUBJECTS` 字典，改为从 `config_store.get_subject_names()` 动态生成映射 | 测试：修改 config 中的科目名称，验证 submit 是否使用新科目 |

| R6 | 可维护性风险 | `core/analyzer.py:116-117` 和 `168-169` | `calculate_total_study_days()` 和 `calculate_study_days()` 两个方法功能完全相同，都是 `return len(set(r.date for r in self.records))` | 代码重复，增加维护成本，调用者可能困惑该使用哪个 | 删除其中一个方法，统一使用一个名称（如 `calculate_study_days()`） | 全局搜索两个方法的调用点，确认替换后测试通过 |
| R7 | 可维护性风险 | `core/analyzer.py:75-87` `find_frequent_problems()` | 高频问题识别使用简单的中文标点分割 + 词频统计，对于连续中文文本（无空格分隔）几乎无法正确分词 | 高频问题识别不准确，"快速排序分区"和"快速排序"会被识别为不同的词 | 引入简单的中文分词（如 jieba）或改用关键词匹配（预定义关键词列表） | 构造包含连续中文的测试数据，验证分词准确性 |

## 4. 建议优先级

### 高优先级

- **R1（ID 碰撞）**：ID 是记录的唯一标识，碰撞会导致数据覆盖或查询错误。虽然概率不高，但在批量导入或高频使用时风险增大。建议立即改用 `uuid.uuid4()`。
- **R3（冒号分隔符）**：这是一个潜在的崩溃点，且触发条件不可控（用户可能输入包含冒号的章节名）。建议立即改用 `split(":", 1)`。

### 中优先级

- **R2（缓存一致性）**：在正常使用场景下影响较小（单用户单进程），但如果用户手动修复数据文件后需要重启程序才能生效。建议尽快添加缓存失效机制。
- **R4（硬编码科目）**：当前配置中科目列表固定，影响不大。但如果后续支持自定义科目，此问题会暴露。建议尽快改为动态读取。
- **R5（备份覆盖）**：备份是为了应对数据损坏，如果备份本身被覆盖就失去了意义。建议尽快改为带时间戳的备份。

### 低优先级

- **R6（代码重复）**：不影响功能，但影响代码可读性和维护性。建议后续重构时统一。
- **R7（中文分词）**：高频问题识别本身是辅助功能，精度要求不高。如果后续需要更准确的分析，可以引入分词库。
- **R8（写入异常）**：正常使用下较少触发，但异常处理是防御性编程的基本要求。建议后续完善。
- **R9（测试路径）**：当前环境下测试可正常运行，但不利于 CI/CD。建议后续整理项目结构。
- **R10（预测公式）**：冲刺预测是参考性质的功能，精度要求不高。建议后续优化预测模型。

## 5. 总结

本次审查覆盖了 408 考研学习助手项目的全部核心代码，包括入口文件、模型层、核心业务逻辑、存储层和测试代码。代码整体结构清晰，分层合理，测试覆盖率达到 100%（160 条测试用例全部通过）。

最大的风险是 **ID 生成碰撞**（R1）和**冒号分隔符解析错误**（R3），这两个问题在特定条件下可能导致数据损坏或程序崩溃。其次是**缓存一致性**（R2）和**硬编码科目映射**（R4），这些问题在正常使用下影响较小，但在异常或扩展场景下会暴露。

建议修复高优先级问题后，重新运行全部测试用例验证无回归。修复完成后，可通过以下方式重新验证：

1. 运行 `pytest test_408_assistant.py -v` 确认全部测试通过
2. 手动执行 `python cli.py submit` 提交一条包含冒号的章节名，验证不崩溃
3. 检查生成的 ID 格式是否为 UUID 或足够长的随机数
4. 验证备份文件是否带有时间戳
