# 小说 RAG 最小 Demo · Token 策略与资料索引

## 1. 文档目标

本文件用于管理小说 RAG 最小 Demo 的项目资料、上下文使用方式和 Token 控制策略。

当前阶段只实现：

```
TXT 输入 → 章节切分 → chunk 切分 → chunks.json 输出
```

因此 AI 不需要读取完整小说全文，也不需要读取完整 chunks.json。

AI 只需要读取：

- 项目需求文档
- 设计文档
- 工作流文档
- 小样本文本
- 错误日志
- 必要的代码片段

---

## 2. 当前阶段边界

### 当前阶段包含

- 读取本地 TXT 小说
- 识别章节标题
- 切分 chunk
- 导出 chunks.json
- 输出统计信息
- 编写测试与质量门禁文档

### 当前阶段不包含

- Embedding
- FAISS
- LLM 问答
- Agent
- MCP
- Playwright
- 自动追更
- 人物关系抽取
- 时间线分析
- 前端页面

## 4. Token 使用分级策略

### L0：禁止提供

包括：

- API Key
- .env 文件内容
- 账号密码
- 手机号
- 邮箱验证码
- 私人聊天记录
- 系统敏感路径
- 完整版权小说全文

处理方式：不得粘贴给 AI，不得上传，不得写入公开文档。

---

### L1：只提供摘要或片段

包括：

- 小说/轮回乐园.txt
- 轮回乐园_chunks.json
- 长日志
- 大型测试输出

处理方式：只提供以下内容之一：

- 前 100 行
- 章节标题样例
- 前 3 条 JSON
- 随机 3 条 JSON
- 报错附近 30 行
- 统计信息

---

### L2：可全文提供

包括：

- product-prd.md
- design-options.md
- rag_flow.md
- dev-workflow.md
- test-strategy.md
- qa-gates.md
- skill-mcp-harness.md
- test-record.md
- learning-record.md
- split.py
- data/sample.txt

处理方式：可以全文提供给 AI，但如果文件很长，优先提供相关章节。

---

### L3：索引优先

包括：

- 多个项目文档
- 多轮对话记录
- 历史版本文档
- 大型代码目录

处理方式：先提供文件索引和摘要，让 AI 判断需要哪部分，再提供具体片段。

---

## 5. 当前阶段推荐上下文包

### 用于生成 split.py 的上下文包

包含：

1. product-prd.md 中的当前阶段目标和 P0 需求
2. design-options.md 中选择方案 B 的理由
3. dev-workflow.md 中 T2 到 T5 的开发任务
4. test-strategy.md 中 TC-001、TC-005、TC-009、TC-012
5. 原始小说 TXT 的前 100 行或章节标题样例

不包含：

- 完整小说全文
- 完整 chunks.json

---

### 用于调试章节正则的上下文包

包含：

1. `detect_chapters()` 函数代码
2. 原始 TXT 中连续 20-50 行章节标题附近文本
3. 当前程序输出的章节数
4. 失败现象，例如"章节数为 1"

不包含：

- 完整小说全文

---

### 用于检查 chunks.json 的上下文包

包含：

1. chunks.json 前 3 条
2. chunks.json 随机 3 条
3. 命令行统计信息
4. json.tool 是否通过

不包含：

- 完整 chunks.json

---

### 用于写测试记录的上下文包

包含：

1. test-strategy.md
2. 实际执行命令
3. 实际命令行输出
4. 失败截图或错误文本
5. 修复后的回归结果

## 6. 敏感数据隔离规则

以下内容禁止交给 AI：

- API Key
- token
- cookie
- .env 文件内容
- 账号密码
- 手机号
- 身份证号
- 邮箱验证码
- 私人聊天记录
- 支付信息
- 系统凭据
- 完整版权小说全文

如果日志中包含敏感数据，必须先打码：

```
API_KEY=sk-****abcd
phone=138****0000
email=x***@example.com
```

## 7. 版本控制策略

### 文档版本号规则

每个文档头部标注版本：

```markdown
> 版本：v1.0
> 日期：2026-06-24
> 状态：当前阶段
```

### 防止 AI 使用过期资料

- 提供文档时检查版本号
- 如果文档有更新，主动告知 AI "请使用 v2.0 版本"
- 如果代码和文档不一致，以代码为准，同时更新文档

### 版本更新触发条件

| 触发事件 | 需要更新的文档 |
|----------|----------------|
| 修改需求 | product-prd.md |
| 修改技术方案 | design-options.md |
| 修改函数签名 | rag_flow.md、dev-workflow.md |
| 修改测试用例 | test-strategy.md |
| 修改 Gate 条件 | qa-gates.md |
| 发现新的敏感数据 | token-strategy.md |

## 8. Token 节省技巧

### 当前阶段具体措施

| 措施 | 说明 | 节省效果 |
|------|------|----------|
| 不提供完整小说全文 | 只提供前 100 行或章节标题样例 | 节省 99% Token |
| 不提供完整 chunks.json | 只提供前 3 条 + 字段结构 | 节省 95% Token |
| 优先提供相关函数 | 修改 detect_chapters() 时只提供该函数 | 节省 50% Token |
| 使用小样本测试 | 用 sample.txt 代替完整小说 | 节省 90% Token |
| 分步骤提供上下文 | 一次只提供一个任务的上下文 | 节省 30% Token |
| 复用已有文档 | 引用已有文档而不是重新描述 | 节省 20% Token |

### 后续阶段 Token 控制

| 阶段 | 主要 Token 消耗 | 控制策略 |
|------|-----------------|----------|
| 阶段 2 Embedding | 模型加载、向量计算 | 不向 AI 提供完整向量数据 |
| 阶段 3 FAISS | 索引构建、检索测试 | 只提供 top-k 结果，不提供完整索引 |
| 阶段 4 RAG 问答 | LLM 调用、回答生成 | 控制 Prompt 长度，限制 chunks 数量 |
| 阶段 5 Agent | 多工具调用、结果整合 | 限制工具调用次数，缓存中间结果 |
