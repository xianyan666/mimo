# 小说 RAG 最小 Demo · 设计方案取舍表

## 1. 项目背景

本项目目标是构建一个面向长篇中文网文的 RAG 最小 Demo。

完整 RAG 系统通常包括：

```
TXT 文本输入 → 文本预处理 → Embedding 向量化 → FAISS 向量索引 → Top-K 检索 → LLM 生成回答
```

但当前阶段只实现第一步：

```
TXT 文本输入 → 章节切分 → chunk 切分 → chunks.json 输出
```

当前设计重点是：如何选择一个可在短时间内跑通、可验收、可扩展的最小实现路线。

---

## 2. 方案对比总表

| 方案 | 方案描述 | 优点 | 风险 | 是否选择 |
|------|----------|------|------|----------|
| A | 一次性实现完整 RAG（从 TXT 到 LLM 问答） | 功能完整，展示效果强 | 范围过大，30 分钟内不稳定，出错点过多 | 否 |
| B | 先实现 TXT 切分与 chunks.json 输出 | 范围清晰，易验收，不依赖外部 API | 当前阶段不能直接问答，chunk 可能语义断裂 | 是 |
| C | 使用 LangChain / LlamaIndex 框架搭建 RAG | 开发速度快，框架功能完整 | 封装过深，不利于学习底层流程，可复现性差 | 否 |
| D | 先做网页爬取或自动追更 | 更接近真实产品形态 | 涉及版权问题，反爬不稳定，任务偏离 RAG | 否 |

---

## 3. 方案 A：一次性实现完整 RAG

### 方案描述

从 TXT 输入开始，一次性完成全部 RAG 链路：

- 读取小说 TXT
- 切分文本
- 生成 Embedding
- 建立 FAISS 索引
- 实现用户查询
- 调用 LLM 生成回答

### 优点

- 功能完整，可直接演示问答效果
- 展示效果强，更接近最终小说 RAG 系统
- 一步到位，不需要分阶段实现

### 风险

- 任务范围过大，单次实现容易遗漏细节
- 调试链路过长，Embedding → FAISS → LLM 任意一环出错都需要重新排查
- 出错点太多，30 分钟 Demo 内不稳定
- 如果前置切分质量差，后续检索和回答都会受影响
- 可能消耗较多模型调用额度，不适合反复调试
- 课程验收时难以快速定位问题所在环节

### 不选择原因

当前课程目标是跑通一个可复现最小 Demo，而不是一次性完成完整系统。方案 A 虽然展示效果强，但风险过高，不适合作为第一阶段实现路线。

---

## 4. 方案 B：先实现 TXT 切分与 chunks.json 输出

### 方案描述

当前阶段只完成文本预处理：

- 读取本地小说 TXT
- 自动识别章节标题
- 按章节切分正文
- 按 chunk_size 和 overlap 切分文本
- 保存 chunks.json
- 输出章节数、chunk 数、平均 chunk 长度等统计信息

### 优点

- 范围清晰，只处理文本，不涉及模型调用
- 容易验收，输出 JSON 可直接检查
- 适合 30 分钟最小 Demo
- 不依赖外部 API
- 不消耗大模型额度
- 输出结果可以作为后续 Embedding 和 FAISS 的输入
- 出错时容易定位，例如章节识别失败、编码失败、chunk 参数不合理

### 风险

- 当前阶段不能直接问答，用户体验不如完整 RAG
- chunk 按字符切分可能造成语义断裂
- 章节标题正则可能不适配所有小说格式
- 没有向量检索能力，无法验证切分质量对检索的影响

### 选择原因

选择方案 B 作为当前阶段实现方案，因为它最符合最小 Demo 的要求，能先验证小说文本是否可以被稳定处理为结构化数据，为后续 RAG 模块打基础。

---

## 5. 方案 C：使用现成框架直接搭建 RAG

### 方案描述

使用 LangChain 或 LlamaIndex 的文档加载、文本切分、向量库封装和问答链能力，快速搭建一个 RAG Demo。

### 优点

- 开发速度快，框架提供开箱即用的组件
- 框架功能完整，覆盖从切分到问答的全流程
- 后续扩展方便，社区资料多
- 可以快速验证 RAG 效果

### 风险

- 框架封装较多，不利于学习底层流程
- 出错时不容易定位，需要深入框架源码排查
- 课程验收时不容易说明每一步细节
- 对中文网文章节结构的定制能力较弱，需要额外适配
- 可能引入较多依赖，降低最小 Demo 的可复现性
- 框架版本更新可能导致 API 变化

### 不选择原因

当前阶段重点是理解 RAG 前置数据处理流程，而不是直接依赖框架完成所有流程。为了保证可控性和可解释性，暂不选择方案 C 作为第一阶段实现方案。

---

## 6. 方案 D：先做网页爬取或自动追更

### 方案描述

使用 Playwright 或其他浏览器自动化工具，从网页中获取小说章节内容，再进行切分和入库。

### 优点

- 更接近真实产品形态
- 后续可以支持自动追更
- 可以减少手动准备 TXT 的过程

### 风险

- 涉及版权和数据来源问题
- 网页结构变化会导致爬取失败
- 反爬机制可能影响稳定性
- 任务重点偏离 RAG Demo
- 30 分钟内很难保证可复现
- 需要额外处理登录、验证码等复杂场景

### 不选择原因

当前阶段已有本地 TXT 文件，不需要先做爬取。为了避免版权、反爬和网页结构变化问题，暂不选择方案 D。

---

## 7. 最终选择

最终选择：**方案 B，先实现 TXT 切分与 chunks.json 输出。**

选择理由：

- 最符合当前最小 Demo 范围
- 可在短时间内实现并验证
- 不依赖云服务和外部 API
- 不消耗 Plus 或 API 额度
- 能为后续 Embedding、FAISS 和 RAG 问答提供稳定输入
- 风险集中在文本读取和切分规则上，便于定位和修复

---

## 8. 当前实现边界

当前实现：

- `load_txt()` — 读取 TXT 文件，UTF-8/GBK 自动检测
- `detect_chapters()` — 正则识别章节标题
- `split_chapter()` — 按 chunk_size 和 overlap 切分章节
- `build_chunks()` — 构建结构化 chunk 数据
- `save_chunks_json()` — 导出 chunks.json
- 命令行运行
- 输出统计信息

当前不实现：

- Embedding
- FAISS
- LLM 问答
- Agent
- MCP
- Playwright 爬取
- 前端界面

## 9. 后续演进路线

### 阶段 2：Embedding 向量化

将 chunks.json 中的 text 转换为向量，使用 text2vec、m3e 或 bge 等中文 Embedding 模型。

### 阶段 3：FAISS 检索

建立 FAISS 索引，支持根据用户问题检索 top-k chunk。

### 阶段 4：RAG 问答

将检索结果和用户问题拼接成 Prompt，调用 LLM 生成回答。

### 阶段 5：Agent 扩展

根据问题类型选择不同工具，例如剧情检索、人物检索、章节定位等。

---

## 10. 风险控制

### 风险 1：章节识别失败

**表现：** 章节数为 1 或明显少于预期。

**控制措施：** 查看原始 TXT 章节标题格式，调整章节正则表达式。

### 风险 2：编码读取失败

**表现：** 乱码或 UnicodeDecodeError。

**控制措施：** 依次尝试 UTF-8、GBK、GB18030。

### 风险 3：chunk 语义断裂

**表现：** chunk 首尾句子被截断。

**控制措施：** 使用 overlap 缓解，默认 chunk_size=1000，overlap=150。

### 风险 4：输出 JSON 过大

**表现：** chunks.json 文件较大，打开较慢。

**控制措施：** 优先保证机器可读，不要求人工完整打开；测试时可抽样检查前 5 条和随机 5 条。

---

## 11. 结论

当前阶段选择方案 B，不是因为完整 RAG 不重要，而是因为文本预处理是 RAG 系统的基础。只有先稳定生成结构化 chunks.json，后续 Embedding、FAISS 和问答模块才有可靠输入。
